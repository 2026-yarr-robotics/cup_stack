프론트엔드 (React + Vite)

SPA로 빌드되어 nginx가 /에 서빙. 명령 입력은 단순 텍스트 파서 → REST 호출 흐름.

CommandInput.submit() → onSend(cmd)
  → App.handleCommand(cmd)
    → parsePickArgs(cmd)          // "pick X Y" / "pick -x X -y Y --cup N" 등 파싱
    → if (!wsConnected || !robotOnline) return
    → pickOne(x, y, opts)         // api.ts:125
      → fetch(`${baseUrl}/api/robot/skill/pick`, POST,
              JSON({x, y, nested_count|cup_top_z, ori?}))

베이스 URL 결정 우선순위: localStorage['cup_api_base'] → VITE_API_BASE_URL → 디폴트 https://yarr-api.simplyimg.com.

FastAPI 서버 + Nginx

3개 도메인이 별도 컨테이너로 분리. Nginx가 prefix로 라우팅:

도메인 객체(robot_domain 등)는 lifespan에서 한 번 생성·주입되고, 라우터는 모듈 전역 참조로 사용 (FastAPI Depends 미사용 — 의도된 패턴).

skill_api 프록시

/api/robot/skill/pick은 서버가 호스트 ROS 2 측의 skill_api_node (http://host.docker.internal:8765/skill/pick)로 단순 프록시. 최초 호출 시 skill_api가 미기동이면 bringup_agent의 /task/start로 lazy spawn하고 /status ready 폴링.

Bringup Agent

Docker 안에서는 호스트의 ROS 2 launch / 프로세스를 직접 제어할 수 없으므로, 호스트에 작은 HTTP 에이전트를 띄워둠. server/bringup_agent.py가 그것.

엔드포인트설명
GET /healthliveness
GET /statusbringup 상태(idle/running/failed). 외부 launch(bringup_real.sh 직접 실행)도 pgrep으로 감지해 노출
POST /start / /stopbringup_real / bringup_sim 시작·종료
POST /task/start / /task/stop임의의 ros2 launch task (예: skill_api)
GET /task/status?name=skill_api해당 태스크의 로그/상태

태스크 시작 명령은 다음 setup.bash들을 순차 source: /opt/ros/humble → (있다면) ~/ws_moveit/install → ~/ros2_ws/install → cup_stack workspace install → ros2 launch cup_stack <cmd>.launch.py.

토픽/네임스페이스

bringup_real.sh가 호출하는 dsr_bringup2_rviz.launch.py는 namespace=/dsr01로 모든 노드를 띄움. 따라서:

  • 관절 상태: /dsr01/joint_states (root /joint_states 아님)
  • 컨트롤러 매니저: /dsr01/controller_manager
  • Doosan 모션 서비스: /dsr01/motion/move_line, movej_h2r, movel_h2r 등

이 때문에 moveit_py.yaml의 joint_state_topic도 /dsr01/joint_states로 맞춰져 있음 (commit b0c03b7).

컨트롤러 관리 원리 (핵심)

이 시스템은 ros2_control 기반이며 두 컨트롤러가 같은 컨트롤러 매니저에 공존한다. 핵심은 command interface 자원 점유 패턴이 서로 다르다는 것이다.

ros2_control 컨트롤러 라이프사이클

┌────────────────────────────────────────────────────────────┐ │ /dsr01/controller_manager │ │ │ │ ┌──────────────────────┐ ┌──────────────────────┐ │ │ │ ResourceManager │ │ ControllerManager │ │ │ │ - hardware ifaces │◀────┤ - load/unload │ │ │ │ - command/state │ │ - configure/activate│ │ │ └──────────┬───────────┘ └──────────┬───────────┘ │ │ │ │ │ │ │ claim/release │ load_controller │ └──────────────┼────────────────────────────┼────────────────┘ │ │ ┌───────┴────────┐ ┌─────────┴──────────┐ │ dsr_controller2│ │ dsr_moveit_ │ │ (Doosan SDK │ │ controller (JTC) │ │ 직통, 자원 │ │ position 인터페이스│ │ 미점유) │ │ 점유 │ └────────────────┘ └────────────────────┘

왜 두 컨트롤러가 충돌 없이 공존 가능한가

ros2 control list_controllers -v --controller-manager /dsr01/controller_manager 실행 결과:

joint_state_broadcaster   joint_state_broadcaster/JointStateBroadcaster   active
    claimed interfaces:
    required state interfaces: joint_1/position, joint_1/velocity, ...
dsr_controller2            dsr_controller2/RobotController                 active
    claimed interfaces:                  ← 비어있음!
    required command interfaces:         ← 비어있음!
dsr_moveit_controller      joint_trajectory_controller/JointTrajectoryController  active
    claimed interfaces: joint_1/position, joint_2/position, ...
    (JTC가 position 인터페이스 점유)

dsr_controller2는 ros2_control의 command interface를 전혀 claim하지 않는다. 대신 Doosan DRFL SDK를 통해 로봇 컨트롤러와 직접 TCP 통신. 반면 dsr_moveit_controller는 표준 JTC로 position command interface를 점유. 두 컨트롤러가 같은 리소스를 두고 다투지 않으므로 동시에 active 가능.

현재 bringup이 spawn하는 컨트롤러

업스트림 dsr_bringup2_rviz.launch.py는 dsr_controller2와 joint_state_broadcaster 두 개만 spawn한다. dsr_moveit_controller는 ros2_controllers.yaml에 정의는 되어 있으나 띄우지 않음.

Doosan 공식 dsr_moveit_config_m0609/launch/demo.launch.py는 셋 모두 spawn하므로, 공식 의도된 구성은 "셋 다 active"이다.

cup_stack의 처방 (commit f949615)

업스트림 launch를 건드리지 않고 skill_api.launch.py에서 dsr_moveit_controller를 추가 spawn한다:

dsr_moveit_controller_spawner = Node(
    package="controller_manager",
    executable="spawner",
    arguments=[
        "dsr_moveit_controller",
        "--controller-manager", "/dsr01/controller_manager",
    ],
)

spawner 노드는 controller_manager 서비스를 호출해 load → configure → activate를 한 번에 수행하고 즉시 종료. 컨트롤러는 컨트롤러 매니저 내부에 등록된 상태로 남는다 (skill_api 노드 수명과 무관).

네임스페이스 정합 (skill_api Node를 /dsr01 namespace에)

MoveIt의 moveit_simple_controller_manager는 controller 이름을 상대명으로 사용해 액션 client를 만든다 (<name>/follow_joint_trajectory). 실제 컨트롤러는 /dsr01/controller_manager 아래에서 활성화되어 액션 서버 경로는 /dsr01/dsr_moveit_controller/follow_joint_trajectory.

moveit_simple_controller_manager는 MoveIt 내부에서 별개의 노드로 생성되므로 launch의 Node(remappings=[...])로는 그 노드의 이름 해석에 영향을 주지 못한다 (실제로 적용해 봤더니 동작 안 함). 가장 안전한 방법은 부모 ROS 노드의 namespace 자체를 /dsr01로 두는 것: 

Node(
    package="cup_stack",
    executable="skill_api_server",
    namespace="/dsr01",          # ← 핵심
    parameters=[...],
)

이러면 내부 moveit_simple_controller_manager의 상대명 dsr_moveit_controller/follow_joint_trajectory이 자연스럽게 /dsr01/dsr_moveit_controller/follow_joint_trajectory로 해석되어 JTC와 연결된다. 별도 remap 불필요.

부수효과: skill_api_node 자체도 /dsr01/skill_api_node가 된다. 외부 HTTP 인터페이스(:8765)와 도메인 로직에는 영향 없음.

Trajectory 실행 경로

skill_api_node (CupStackRuntime) │ arm.plan(...) ─────────────────▶ MoveItCpp (planning) │ (Pilz PTP/LIN → 실패 시 OMPL) │ robot.execute(robot_trajectory, blocking=True) │ │ │ ▼ trajectory_execution_manager │ → moveit_simple_controller_manager │ → ActionClient(/dsr_moveit_controller/follow_joint_trajectory) │ ↳ remap ⇒ /dsr01/dsr_moveit_controller/follow_joint_trajectory ▼ JointTrajectoryController (dsr_moveit_controller) │ position command stream (control_msgs/FollowJointTrajectory) ▼ ros2_control ResourceManager │ joint_N/position command interface ▼ dsr_hardware2 (SystemInterface) ▼ Doosan DRFL ──TCP:12345──▶ M0609 Robot Controller

반환 경로 — 컨트롤러는 실시간 trajectory feedback을 액션 클라이언트로 회신. 성공/실패는 robot.execute()의 bool 반환값으로 전달되며 (commit f949615) 이전엔 반환값을 무시해 ABORTED를 success로 위장하는 버그가 있었다.

그리퍼 경로 (별도)

RG 그리퍼는 MoveIt 경로를 거치지 않고 별도 토픽/서비스(gripper_node)로 제어된다. 그래서 trajectory 실행이 실패해도 그리퍼는 동작했고, 이는 과거 진단을 혼란시킨 부수 증상.

픽 명령의 end-to-end 흐름

1. 사용자: "pick 0.45 0.12" 입력 (또는 curl POST)
2. Frontend: parsePickArgs → pickOne(x=0.45, y=0.12, {nestedCount:1})
3. Frontend: POST {baseUrl}/api/robot/skill/pick
              body: {"x":0.45, "y":0.12, "nested_count":1}
4. Nginx: → robot:8001 컨테이너
5. FastAPI (server/server/routers/robot.py:175): skill_pick(body)
6. RobotDomain.pick_skill (server/server/domains/robot.py:541)
     - skill_api ready 체크 (8765/status)
     - 미기동이면 bringup_agent /task/start로 lazy spawn → ready 폴링
     - HTTP POST http://host.docker.internal:8765/skill/pick
7. skill_api_node (PickCupSkill, ros2-cup-stack):
     [1] approach @ PICK_SAFE_Z (0.55 m)
         → runtime.try_move_to_pose → MoveItPy plan + execute
     [2] gripper OPEN  (별도 그리퍼 토픽)
     [3] descend → z=pick_z_base (0.323 m)
     [4] GRIP
     [5] lift → PICK_SAFE_Z
     각 step의 execute() 결과를 확인 (commit f949615) — ABORTED면 즉시 False 반환.
8. 응답 본문: {"success": bool, "skill": "pick", "detail": "..."}
9. Frontend: 로그에 "Pick complete" / "Pick failed" 표시.
